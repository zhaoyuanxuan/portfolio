          A · Thesis

          不是页面优化 · 而是组织决策系统的重构

          面对一套随业务规模化逐渐失效的人才管理系统，我把切入点从 UI 抛光重新放回信息流与决策流：管理者真正需要看到什么、理解什么、决策什么。覆盖绩效 / 盘点 / 晋升等核心 OD 场景，把"功能集合"重塑为"统一决策平台"。

          3 类 Core scenarios 绩效评价 · 人才盘点 · 晋升校准
          4 角色 Stakeholders 普通员工 · 线上管理者 · BP / OD · 高管
          v.4 Iterations 第四版评价流程 · 完整端到端
          1 套 Unified IA 跨场景共享层级 · 状态语言 · 反馈规则
          B · Diagnosis

          流程驱动型系统的三重病

          界面复杂只是表象。真正的问题是组织在规模化后，信息流和决策流开始失效。

          P · 01 · 信息散落

          关键决策信息淹没在低价值内容中

          页面结构不一致，信息分散在不同模块与表格里，相似场景缺乏统一表达——管理者完成一次绩效判断，需要跨页面切换、比对、记忆。

          Symptom · 跨页比对 · 高记忆负载
          P · 02 · 流失

          组织的信息流 / 决策流开始失效

          表面是操作成本高，本质是系统缺少面向"组织决策"的信息架构与行为逻辑——继续做 UI 抛光只能延缓崩坏，不能解决系统命题。

          Symptom · 决策无依据 · 偏差不可见
          P · 03 · 不确定

          复杂系统下的行为不确定性

          "筛选会影响哪些区域？""当前状态意味着什么？""下一步该做什么？"——这些过去高度模糊的问题，让管理者在高密度信息里反复迷失方向。

          Symptom · 反馈缺失 · 路径模糊
          C · Reframe

          从流程驱动型系统 → 组织决策平台

          不再继续堆叠历史流程；而是从业务目标重新反推系统结构。

          Before · 流程驱动

          系统是功能的集合

          • 页面 / 表格按历史功能堆叠
          • 同一类信息在多处重复、口径不一
          • 状态语言、操作规则散落在每个模块各自定义
          • 关键信息埋在二级 / 三级页面
          • 无法横向支撑"组织级"判断
          →
          After · 决策驱动

          系统承载组织认知 · 标准 · 决策逻辑

          • 以「组织校准 / 风险识别 / 绩效判断 / 人才发展」为信息组织依据
          • 统一的信息层级、状态语言、交互规则
          • 围绕"人"与"组织"重组视图，而非围绕"表单"
          • 结果分布、风险提示、历史趋势、评价偏差前置
          • 跨绩效 / 盘点 / 晋升场景共享同一套设计基础
          D · Workflow

          第四版评价流程 · 5 个阶段 · 端到端

          把绩效评价从"任务列表 + 表格"重构为"准备 → 评价 → 提交校对 → 完成"的清晰节奏；每个阶段都有明确的角色 · 信息密度 · 关键决策。

          F · Principles

          7 条贯穿全局的设计原则

          不是局部体验优化的细则，而是跨绩效 / 盘点 / 晋升场景的系统性约束——后续业务扩展时仍然有效。

          01

          围绕「人」与「组织」重组信息

          不是围绕表单和模块。同一个人 / 同一个组织在哪个流程里，看到的都是统一视图。

          02

          提升信噪比 · 不是展示更多

          设计目标从"补全字段"转为"前置关键信息"——结果分布、风险提示、历史趋势、评价偏差。

          03

          渐进披露 · 优先处理关键任务

          主表负责扫读群体异常 / 档案承接明细查询 / hover 提供轻量增量信息。

          04

          一致性反馈机制

          筛选影响哪些区域、当前状态是什么、下一步是什么——全场景使用同一套反馈语言。

          05

          异常前置 · 决策导向排序

          把"需要被处理的差异"从表格中段拉到顶部，让管理者第一眼看到风险。

          06

          从个体打分 → 组织级校准

          校准不是结果展示，而是组织层面对齐口径、暴露偏差的核心动作——单独建模为流程节点。

          07

          为可持续演进而设计

          同一套层级 · 状态 · 反馈规则跨绩效 / 盘点 / 晋升复用，业务扩张时不需要从零再设计一次。

          A · Thesis

          不是面试工具上线 · 而是把"算法刷题"重新定义为"AI 协同问题解决能力"

          AI 工具普及后，传统 LeetCode 风格面试逐渐偏离真实工作场景。本项目以快手社招为 MVP，把"是否能调度 AI、结合工程上下文解决真实问题"建模为一套可考核、可观察、可回放的招聘基础设施——覆盖招聘系统 · 萝卜面试间 · 自研 IDE · 评估与回放四条线。

          4 维度 Evaluation axes Decomposition · Orchestration · Verification · Production
          2 模式 Sync & Async 面试中实时考核 + 面试前异步考核（60min）
          3 角色 Stakeholders 招聘 HR · 面试官 · 候选人，权限按轮次隔离
          6 层 System layers IDE · AI 调度 · 观测 · 评估 · 面试官端 · 题库
          B · Diagnosis

          AI 时代下技术面试的三重失效

          竞品调研（HackerRank / Codility / 牛客 / Meta）后我们发现：问题不只是"加一个 AI 聊天框"。真正的失效在题目、信号和招聘流程三个层面同时发生。

          P · 01 · 题目失效

          LeetCode 风格与真实工程已经脱节

          单文件 · 无依赖 · 无上下文的算法题，在 AI 工具普及后几乎可以被秒杀，无法区分候选人在多文件工程里的判断力与协作能力。

          Symptom · AI 一键解题 · 区分度坍塌
          P · 02 · 信号失效

          "AI 能否解" ≠ "人能否驾驭 AI"

          仅看最终提交无法回答关键问题：是否在采纳 AI 输出前做了验证？是否能识别幻觉 API？Prompt 是否结构化？面试官只看代码 diff，看不到决策过程。

          Symptom · 决策过程不可见 · 信号丢失
          P · 03 · 流程失效

          招聘流程从未为"AI 考核"建模

          从 R 安排面试 → 面试官出题 → 候选人异步答题 → 下一轮面试官查看 IDE，整条链路涉及多角色 · 多状态 · 多通知。任意一环失稳，整个考核都会失败。

          Symptom · 多角色协同 · 状态易飘移
          C · Reframe

          从"考代码" → "考人机协作"

          把考核对象从"代码本身"重定义为"候选人 × AI"组成的临时协作系统。系统的评估单位不是答案，而是过程信号。

          Before · 考代码

          考核 = 提交了什么

          • 题目：单文件算法 · 与真实工程无关
          • 环境：禁用 AI · 假设候选人裸手写代码
          • 评判：通过率 + 复杂度 + 风格
          • 流程：候选人独自完成 · 面试官只看结果
          • 数据：一个分数 · 无法回放
          →
          After · 考人机协作

          考核 = 如何与 AI 共同抵达

          • 题目：多文件 · 含 AI 陷阱的真实工程
          • 环境：开放 AI · 强制结构化 Prompt
          • 评判：4 维度 · 规则 + LLM-as-judge · 证据回链
          • 流程：考核与面试解耦 · 下一轮面试官可读 IDE
          • 数据：Timeline · Decision Points · 全程可回放
          D · Workflow

          端到端流程 · 5 阶段 · 跨 4 个系统

          从 R 安排面试到下一轮面试官读 IDE，整条链路在招聘系统 · 萝卜 · 自研 IDE · 通知中心之间穿行。每个阶段都有清晰的角色 · 触发器 · 状态机。

          PHASE · 01

          安排面试

          • R 在安排/修改面试时勾选「AI coding」
          • 招聘系统同步萝卜：生成考核单据
          • 默认无值 · 必填 · 灰度面试官范围
          PHASE · 02

          面试官出题

          • KIM 通知 → 进入题库选 1 题
          • "我收藏的"优先 · README 预览
          • 下发后不可撤回 · 长时间未出题双向催
          PHASE · 03

          候选人答题

          • 考核须知 → 进入 IDE · 60min 倒计时
          • 双机位监控 · 离开 2min 内可重入 · 最多 2 次

          PHASE · 04

          考核结束

          • 主动提交 / 系统自动提交 · 容器冻结

          PHASE · 05

          下一轮面试官

          • 面试间新增「AI coding 考核」tab
          • 只读 IDE · 可见 reference / 答案文件夹
          • 行为摘要弹窗 · 用于面试现场追问
          E · Architecture

          六层系统骨架 · 从候选人到决策的全链路

          不止是一个嵌入萝卜的 IDE。整套系统沿"采集 → 调度 → 观测 → 评估 → 呈现"的方向被刻意切分为 6 层——每一层都有自己的状态机和数据契约。

            设计方案落地在 7 个面向不同角色的核心界面上——每一屏只承担一个意图，跨屏之间通过统一的视觉语言与状态机彼此衔接。下方为各核心界面的结构原型。

            H · Metrics

            MVP 灰度目标 · 让结果可被业务衡量

            在量化目标之外，更重要的是"考核结果是否真的影响了招聘决策"——这是衡量这套系统价值的唯一定性指标。

              I · Principles

              7 条系统设计原则

              不是局部体验细则，而是这套系统在演进中始终不能让步的边界。

              01

              观察过程而非结果

              Observability 是评估的唯一事实底座；所有评分点都必须能回链到具体的 Timeline 事件。

              02

              AI 调用必须可控

              所有 AI 请求经统一 Proxy · 强制结构化 Prompt · 注入检测 · 幻觉 API 校验。

              03

              题目必须防一键秒杀

              每道题都打 AI-Solvability 分数，理想区间是"AI 可 assist 但需人判断"。

              04

              角色权限按轮次隔离

              只有"下一轮"面试官可见 IDE · reference 文件夹仅面试官可见 · 候选人不可二次进入。

              05

              考核与面试解耦

              独立考核房间 · 与面试间互不干扰 · 容器与状态自治 · 取消面试时正确级联失效链接。

              06

              基础设施可观测

              容器启动成功率 · 启动时长 · 中断率作为一等指标，决定系统能否承载招聘信任。
